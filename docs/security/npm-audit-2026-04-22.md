## What npm audit checks

`npm audit` compares the exact package versions in `package-lock.json` against npm's security advisory database. The important detail is that it audits the resolved tree, not only the direct packages listed in `package.json`.

For LAICA, the lockfile currently contains:

| Dependency category | Count |
| --- | ---: |
| Production dependencies | 643 |
| Development dependencies | 154 |
| Optional dependencies | 194 |
| Peer dependencies | 8 |
| Total locked packages | 917 |

That means a security issue could appear in a package LAICA never imports directly. For example, a build tool, UI library, or SDK can pull in smaller helper packages. npm calls those transitive dependencies.

## How to read severity

Security advisories usually have a severity label:

- **Critical:** Usually means an attacker may be able to run code, steal sensitive data, or fully compromise a system in realistic conditions.
- **High:** Serious issue with meaningful exploit potential, often needing quick action.
- **Moderate:** Real issue, but it usually needs narrower conditions, user interaction, development-only access, or a less sensitive target.
- **Low:** Limited impact or difficult to exploit. Still worth tracking, but usually not an emergency.
- **Info:** Advisory information without a clear direct vulnerability.

Severity is not the whole decision. The real impact depends on whether the affected package runs in production, handles user input, processes secrets, executes during builds, or only exists in tests.

## Production vs development impact

LAICA has both runtime packages and build/test packages.

Runtime packages matter most for deployed behavior. Examples include:

- Express and middleware for server routes
- Firebase and API SDKs
- database and session libraries
- file upload and image conversion packages
- frontend runtime packages shipped through the app bundle

Development packages matter too, but the risk is different. Build tooling vulnerabilities often affect the developer or CI environment instead of the deployed user-facing app. That still matters because developer machines and CI can hold credentials, deployment tokens, and encrypted environment keys.

## Recommended next step

Run this in Replit or another networked environment:

```bash

```

If it reports vulnerabilities, use:

```bash
npm audit --json
```

Then review each advisory by asking:

1. Is the affected package used in production or only during development?
2. Is the vulnerable code path reachable in LAICA?
3. Is the fix a patch/minor update or a breaking major update?
4. Does `npm audit fix` change only the lockfile, or does it also change top-level dependency versions?
5. After applying a fix, do `npm run check` and `npm run build` still pass?
