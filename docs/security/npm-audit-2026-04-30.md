## Deep dive: why "zero vulnerabilities" does not always mean "nothing to think about"

### 1. npm audit is a database comparison, not a source-code proof

`npm audit` does not read all source code and decide whether the app is safe. It compares package versions from `package-lock.json` against a security advisory database.

That means a clean result can mean one of several things:

- the package tree is actually clean
- the issue was fixed and the advisory database agrees
- the advisory is not present in the local cache
- the advisory exists, but the environment did not successfully ask the live service

### 2. Production risk and development risk are different

A production vulnerability means real users or public traffic may be able to hit the dangerous code path.

A development-only vulnerability usually means the danger is to:

- a developer machine
- a CI runner
- a preview environment
- local secrets loaded during development

This matters because "dev-only" is lower risk than "internet-facing production", but it is not the same as "safe enough to ignore forever".

### 3. Why the old `esbuild` chain still matters conceptually

The historical remaining issue was tied to an `esbuild` advisory where a malicious website could talk to a developer's local dev server and read responses in certain conditions.

Impact in practical terms:

- A normal LAICA end user on the production app is not the main target.
- A developer running local tooling could be at risk if they simultaneously run a vulnerable dev server and browse to a malicious site.
- If that dev environment holds secrets, database URLs, or source maps, the blast radius is larger than a simple frontend bug.

This is why security reports often separate:

- user-facing runtime risk
- developer workstation risk
- CI/build pipeline risk
